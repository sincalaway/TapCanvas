# TapCanvas AI 创作生产架构

## 目标

把 AI 创作从“写一段 prompt 然后碰运气生成”改造成“有状态、有分层、有准入条件的生产流程”。

核心结论：

- prompt 不是创作核心，prompt 只是表达层
- 创作核心是状态机、证据链、连续性和阶段化生产
- agents-cli 的职责不是写一段漂亮文案，而是判断当前处于什么阶段，并输出下一步可执行动作
- 画布的职责不是堆节点，而是承载创作事实、约束、锚点、执行和结果

这份文档用于统一：

- agents-cli 的编排逻辑
- TapCanvas 画布节点语义
- AI 创作的准入与失败规则
- 后续前后端改造方向

## 一、为什么 prompt 不是核心

如果把 prompt 当作核心，会天然出现以下问题：

- 把“世界设定、角色锁定、镜头设计、动作规划、连续性修复、视频运动”全部塞进一次生成
- 没有阶段边界，失败后无法定位到底是设定错、分镜错、还是执行错
- 一旦结果漂移，只能继续堆 prompt，而不是回到正确层级修复
- 项目状态未读取时，也容易假装理解“当前进度”

正确范式应该是：

- 先确定事实来源
- 再锁定世界与角色
- 再生成关键帧
- 再做镜头扩展
- 再做视频规划
- 最后才做视频执行

因此：

- prompt 属于表达层
- 状态机属于决策层
- 节点分层属于生产结构层
- agents-cli 属于调度层

## 二、生产总原则

### 1. 证据优先

- 没有读取到的项目状态，不得声称已知
- 没有读取到的角色卡、场景图、已有镜头，不得假设存在
- 没有明确来源的剧情、台词、镜头，不得编造

### 2. 连续性优先

- 视频模型不是世界构建器，而是运动执行器
- 世界事实必须先在静帧层稳定
- 任何连续性问题优先回退到关键帧层修复

### 3. 单变量推进

- 每一步只解决一个核心问题
- 不允许在一个动作里同时改世界、改角色、改镜头、改动作

### 4. 显式失败

- 缺关键输入就报错
- 缺上一阶段锚点就中止
- 不允许静默降级、自动补默认、自动改走别的路径

### 5. 结果持久化

- 已生成资产必须保留
- 后处理只能新增记录，不能覆盖或丢弃既有结果
- 创作链路必须可回溯到“证据 -> 计划 -> 执行 -> 结果”

## 三、AI 创作状态机

下面的状态机适用于从小说文本、镜头意图或局部素材出发，生成可持续扩展的 TapCanvas 工作流。

## 状态 1：`source_understanding`

目标：

- 明确当前创作基于哪些真实输入
- 明确当前不是“项目实际进度”，还是“已读取画布后的项目阶段”

输入：

- 小说章节
- 文本片段
- 用户口述目标
- 项目内已有素材索引

输出：

- `novelDoc` / `text` 类证据节点
- 场景范围说明
- 事实清单
- 缺口清单

准入条件：

- 至少读取一种可信来源

准出条件：

- 明确“本轮基于什么证据”
- 明确“缺什么”
- 若未读取画布状态，必须显式声明“不代表项目实际进度”

失败条件：

- 用户要求基于当前项目状态判断，但系统未读取画布/资产状态
- 文本不存在却继续规划

## 状态 2：`constraint_definition`

目标：

- 把风格、禁项、角色连续性、环境连续性变成显式约束

输入：

- 已确认文本
- style bible
- 已确认角色卡
- 已确认参考图

输出：

- `text` 类约束节点
- 统一视觉约束
- 连续性约束
- 禁止项

准入条件：

- 已完成 `source_understanding`

准出条件：

- 角色、环境、风格、禁项至少各有一类显式描述

失败条件：

- 还没定义不变项就直接开始扩镜头

## 状态 3：`world_anchor_lock`

目标：

- 锁定世界事实，让后续镜头共享同一个场景现实

输入：

- 证据节点
- 约束节点

输出：

- 第一张权威基底帧
- 场景地点、时间、空间关系、色调锚点

对应节点：

- `storyboardShot` 基础锚帧
- 必要时补 `image` / `storyboardImage`

准入条件：

- 已存在证据与约束

准出条件：

- 可以明确回答“这是谁、在哪里、何时、周围有什么、整体是什么质感”

失败条件：

- 基底帧无法稳定世界事实
- 画面同时承担过多叙事动作导致无法作为锚点

## 状态 4：`character_anchor_lock`

目标：

- 锁定主体身份连续性

输入：

- 权威基底帧
- 角色约束

输出：

- 可复用的人物关键帧
- 稳定的人物外观、服装、伤痕、姿态基准

对应节点：

- `storyboardShot`
- 必要时角色专属节点或参考图节点

准入条件：

- 已有场景世界锚点

准出条件：

- 后续镜头可基于该人物状态持续派生

失败条件：

- 角色身份漂移
- 服装、发型、伤痕位置频繁变化

## 状态 5：`shot_anchor_lock`

目标：

- 把关键叙事时刻锁成可确认的关键静帧

输入：

- 世界锚点
- 角色锚点
- 导演脚本或镜头计划

输出：

- 一组关键帧
- 每帧只承担一个明确叙事功能

典型关键帧：

- 建立局势
- 人物中景
- 情绪近景
- 爆点前夕

准入条件：

- 世界与角色已基本稳定

准出条件：

- 至少有一张可作为视频上游的关键帧
- 每个关键帧都有清晰用途，不是随机补图

失败条件：

- 关键帧之间世界事实矛盾
- 一张图承担多个不兼容目标

## 状态 6：`single_variable_expansion`

目标：

- 在不破坏事实的前提下扩展镜头

输入：

- 已确认关键帧

输出：

- 侧面
- 反打
- POV
- 景别变化
- 道具状态变化

规则：

- 每步只改一个变量
- 必须显式写出不变项

准入条件：

- 至少存在一张稳定锚帧

准出条件：

- 扩展镜头仍然维持同一世界、同一角色、同一事件

失败条件：

- 同时更改多个变量
- 扩展后无法回溯来源锚帧

## 状态 7：`approved_keyframe_selection`

目标：

- 从候选关键帧里挑出真正可进入视频层的帧

输入：

- 已生成关键帧集合

输出：

- 被明确批准的视频上游关键帧

准入条件：

- 关键帧已存在

准出条件：

- 至少一张关键帧被标记为“可进入视频”

失败条件：

- 没有审批/确认机制就直接推进到视频

## 状态 8：`video_plan`

目标：

- 把“关键帧集合”翻译成“可执行的视频规划”

输入：

- 已批准关键帧
- 台词
- 剧情拍点

输出：

- `composeVideo` 节点
- `storyBeatPlan`：给人看的剧情拍点数组
- `videoPrompt`：给模型执行的高密度运动提示词

关键原则：

- `prompt` 可以保留一句概述
- 但 `prompt` 不能代替 `videoPrompt`
- `storyBeatPlan` 不是装饰字段，而是视频规划的可读结构
- 若上游镜头含台词或字幕约束，视频层必须保留该约束，不能丢词

准入条件：

- 至少有一个通过确认的关键帧

准出条件：

- 视频节点同时具备 `storyBeatPlan` 与 `videoPrompt`
- 能清楚说明每段时长、动作、镜头运动、连续性约束和禁止项

失败条件：

- 只有一句摘要，没有可执行的视频分段
- 视频节点没有承接上游台词/关键叙事互动

## 状态 9：`video_execution`

目标：

- 让视频模型只负责运动表达

输入：

- 已确认的 `composeVideo`
- 已批准关键帧

输出：

- 视频任务
- 视频结果
- 执行日志

准入条件：

- `video_plan` 完整
- 关键帧存在且可信

准出条件：

- 生成结果与日志被完整记录

失败条件：

- 让视频模型承担补世界、补角色、补剧情理解
- 视频失败后不回溯到上游关键帧

## 状态 10：`result_persistence`

目标：

- 保留所有结果与生产痕迹

输入：

- 图片
- 视频
- 日志
- 上下游引用关系

输出：

- 节点结果
- 本地元数据
- 可追溯日志

准入条件：

- 已有执行结果

准出条件：

- 后续续写能读取上一个稳定结果
- 结果与来源链条可回放

失败条件：

- 只保留最终资产，不保留中间锚点与上下游关系

## 四、画布节点分层语义

TapCanvas 画布不应该只按“节点类型”理解，更应该按“生产语义层”理解。

建议统一分成六层。

### 1. 证据层 `evidence`

用途：

- 存放事实来源

典型节点：

- `novelDoc`
- `text`
- 外部素材引用节点

回答的问题：

- 我们基于什么创作
- 哪些内容是确定事实

### 2. 约束层 `constraints`

用途：

- 存放不变项与禁止项

典型节点：

- 视觉设定
- 连续性约束
- 风格规则

回答的问题：

- 后续所有生成都必须遵守什么

### 3. 锚点层 `anchors`

用途：

- 锁世界、锁角色、锁关键场面

典型节点：

- 基础锚帧
- 角色锚帧
- 场景锚帧
- 已批准关键帧

回答的问题：

- 后续镜头到底以谁为准

### 4. 扩展层 `expansion`

用途：

- 在锚点基础上单变量派生

典型节点：

- 分镜关键帧
- POV
- 反打
- 近景/远景变化

回答的问题：

- 在不破坏世界事实的前提下，如何增加镜头覆盖

### 5. 执行层 `execution`

用途：

- 把已确认的上游结构变成运行任务

典型节点：

- `composeVideo`
- 真正的图片/视频执行节点

回答的问题：

- 具体让模型执行什么

### 6. 结果层 `results`

用途：

- 保存产物、日志与占位说明

典型节点：

- 结果节点
- 缺口说明
- 日志节点

回答的问题：

- 生成了什么
- 缺了什么
- 为什么失败

## 五、节点之间的硬依赖

以下依赖应该被当作系统级规则，而不是“推荐”。

### 规则 1

- 没有证据层，不能开始正式创作规划

### 规则 2

- 没有约束层，不能开始关键帧扩展

### 规则 3

- 没有基础锚帧，不能直接创建视频节点

### 规则 4

- 没有批准关键帧，不能进入视频执行

### 规则 5

- 视频节点必须同时包含 `storyBeatPlan` 与 `videoPrompt`

### 规则 6

- 文本类节点不应携带运行态字段

文本类节点包括：

- `text`
- `novelDoc`
- `scriptDoc`
- `storyboardScript`

这些节点表达的是事实和规划，不是任务执行态。

### 规则 7

- 若未读取项目画布/资产状态，禁止输出“当前进度”“当前状态判断”

### 规则 8

- 若上游镜头含台词/字幕约束，下游 `composeVideo` 必须保留相应交互约束

## 六、agents-cli 的职责重构

agents-cli 不应再被理解成“帮我写 prompt 的助手”，而应是生产编排器。

## agents-cli 必须回答的核心问题

### 1. 当前阶段是什么

必须基于证据判断当前处于：

- 从零起场景
- 扩已有镜头
- 修连续性
- 准备进视频
- 视频失败回修

### 2. 当前缺什么

例如：

- 缺项目状态读取
- 缺角色卡
- 缺基础锚帧
- 缺已批准关键帧
- 缺上游台词传递

### 3. 下一步动作是什么

输出应该是：

- `create_base_frame`
- `lock_character_identity`
- `lock_environment_anchor`
- `change_camera_angle`
- `prepare_video_keyframe`
- `generate_video_motion`

而不是空泛建议。

### 4. 应该创建什么节点

agents-cli 的主要产物应当是：

- 画布工作流计划
- 节点配置
- 依赖边

不是一篇创作说明文。

## agents-cli 输出契约

至少应包括：

- `action`
- `reason`
- `nodes`
- `edges`

对视频节点额外要求：

- `config.storyBeatPlan`
- `config.videoPrompt`

对事实类节点额外要求：

- 禁止混入运行态字段

对状态判断额外要求：

- 必须附带“本轮读取了哪些证据”
- 若未读取项目画布状态，必须明确说明不能代表真实项目进度

## 七、提示词在整个系统中的位置

提示词依然重要，但它的职责被收缩为“表达”。

提示词负责：

- 把当前阶段目标表达给模型
- 把不变项和可变项表达清楚
- 把镜头动作和禁止项表达清楚

提示词不负责：

- 判断当前创作阶段
- 判断项目当前进度
- 决定是否能跳过关键帧
- 替代状态机

换句话说：

- 没有状态机，再强的 prompt 也只是锦上添花
- 有了状态机，prompt 才真正稳定可控

## 八、失败策略

### 1. 缺证据就失败

- 未读取画布状态，就不能做项目进度判断
- 未读取文本，就不能假装抽取剧情

### 2. 缺锚点就失败

- 没有基础锚帧，不能强行进视频
- 没有上一段尾帧，章节续写不能静默继续

### 3. 缺连续性就回退

- 视频漂移，回到关键帧
- 关键帧漂移，回到世界锚点或角色锚点

### 4. 缺日志就补日志，不猜测

- 拿不准根因时，先记录输入摘要、分支判断、节点配置和执行结果
- 禁止只返回“生成失败，请重试”

## 九、前端与画布改造建议

### 1. 节点元数据增加“生产语义层”

建议增加统一字段，例如：

- `productionLayer: evidence | constraints | anchors | expansion | execution | results`

价值：

- 便于前端自动分组
- 便于 agents 输出校验
- 便于执行器判断哪些节点可运行、哪些只是说明

### 2. 节点元数据增加“阶段状态”

建议增加统一字段，例如：

- `creationStage`
- `approvalStatus`
- `sourceEvidence`

其中：

- `creationStage` 用于表明节点属于哪个阶段
- `approvalStatus` 用于区分候选关键帧和已批准关键帧
- `sourceEvidence` 用于追踪节点依赖了哪些事实来源

### 3. 画布布局按依赖阶段展开

布局原则：

- 从左到右表达生产阶段
- 同组节点内部再按局部逻辑布局
- 不允许后置自动排版破坏阶段顺序

### 4. 前端创建占位节点后再异步执行

当用户触发创作时：

1. 先创建 queued/running 占位节点
2. 再调用 agents pipeline
3. 完成后回填结果或失败日志

不能等 pipeline 完成后再补节点。

## 十、agents-cli 改造建议

### 1. 从“生成回答”改成“生成生产动作”

默认输出目标应变成：

- 当前阶段
- 缺口
- 下一动作
- 节点计划

### 2. 增加阶段判定器

输入：

- 是否读取了画布
- 是否读取了资产
- 是否存在基础锚帧
- 是否存在已批准关键帧
- 是否存在视频节点

输出：

- sceneType
- workflowState
- nextAction

### 3. 增加计划校验器

校验至少包括：

- 视频节点是否具备 `storyBeatPlan` 与 `videoPrompt`
- 文本节点是否混入运行态字段
- 新场景是否先建基础锚帧
- 是否存在无来源的“当前进度”描述

### 4. 增加状态读取声明

每次 agents 输出都要显式记录：

- 读取了哪些工具
- 读取了哪些真实来源
- 哪些判断可以做
- 哪些判断不能做

## 十一、分阶段落地路线

### 第一阶段：协议收紧

- 固化视频节点双层字段：`storyBeatPlan` + `videoPrompt`
- 禁止文本类节点混入运行态
- 禁止未读画布状态时输出“当前进度”

### 第二阶段：阶段显式化

- 引入 `productionLayer`
- 引入 `creationStage`
- 引入关键帧审批状态

### 第三阶段：执行门禁化

- 无批准关键帧时禁止视频执行
- 无证据节点时禁止生成完整工作流
- 连续性失败时强制回退上游层级

### 第四阶段：元数据闭环

- 把关键帧、尾帧、视频上游引用写入本地元数据
- 后续续写必须读取上一稳定结果

### 第五阶段：agents 成为默认生产入口

- 用户发起章节分镜/镜头续写时，优先走 agents-cli 自动编排
- 人工主要做审核与微调，而不是手工拼装主流程

## 十二、最终结论

TapCanvas 的 AI 创作体系应该建立在以下四层之上：

- 证据链
- 状态机
- 节点语义分层
- agents-cli 编排

prompt 的地位应该被重新定义为：

- 重要
- 但不是核心
- 只能在正确生产结构之上放大效果

如果这套架构成立，那么系统将从“一次性生成工具”升级为“可持续生产系统”。
